# Transcriber GUI PRD

## 1. 产品概述

Transcriber 当前是一个 Python CLI 工具，支持 B 站和抖音视频链接下载、音频提取、Whisper 转写，并生成 Markdown 校对稿。GUI V1 的目标是把它升级为一个 Windows 桌面内容工作台，让用户不用记命令就能批量处理视频链接、查看进度、定位输出文件，并为后续“文案清洗、摘要、标题提取、内容改写”留下产品空间。

## 2. 背景与问题

当前命令行版本已经具备核心能力，但使用体验存在明显短板：

- 用户需要记住命令参数，例如 `--cookies-from-browser chrome`。
- 长任务缺少清晰阶段反馈，用户容易误判为卡死。
- B 站 412、ffmpeg 缺失、Whisper 模型问题等错误不够可操作。
- 输出文件生成后需要用户自己到目录中查找。
- 批量链接处理时，单个失败任务的状态不够直观。
- 未来内容加工能力缺少统一入口。

GUI V1 不应重写核心算法，而应把已有能力产品化。

## 3. 目标用户

主要用户是经常从 B 站/抖音视频中提取口播文案、课程内容、选题素材或复盘内容的个人创作者、运营者或知识整理者。

用户特征：

- 熟悉复制视频链接，但不一定熟悉命令行。
- 关心结果文件在哪里、是否成功、失败后如何重试。
- 能接受本地处理时间较长，但需要状态可见。
- 后续可能希望把转写稿加工成摘要、标题、脚本或图文内容。

## 4. 产品定位

GUI V1 定位为“视频文案工作台”的基础版：

- V1：可视化下载转写、进度和文件管理。
- V2：文案预览、复制、历史记录、失败重试增强。
- V3：AI 摘要、标题提取、图文/脚本生成。

本 PRD 只定义 V1 必须完成的范围，并为 V2/V3 保留结构。

## 5. V1 产品目标

1. 用户可以通过桌面 GUI 粘贴 B 站或抖音链接并创建任务。
2. 用户可以按平台使用专属输入页：抖音、Bilibili。
3. 用户可以看到每个任务的阶段状态和进度。
4. 用户可以选择输出目录，并在完成后打开文件或定位文件夹。
5. 用户可以在 B 站 412 场景下启用 Chrome/Edge cookies 重试。
6. GUI 不阻塞，下载和转写在后台执行。
7. 现有 CLI 继续可用，GUI 与 CLI 共用核心处理逻辑。

## 6. 非目标

V1 不做：

- 多任务并发转写。
- 内置富文本编辑器。
- AI 摘要、标题生成、脚本生成。
- 云同步、账号系统、素材库。
- 跨平台优先适配。第一版以 Windows 桌面为主。
- 对 Whisper 进度做帧级精确 UI。第一版允许阶段级进度。

## 7. 核心用户流程

### 7.1 B 站流程

1. 用户打开 Transcriber GUI。
2. 进入 `Bilibili` tab。
3. 粘贴 `b23.tv` 或 `bilibili.com/video/BV...` 链接。
4. 可选开启 `使用 Chrome Cookies`。
5. 点击 `解析链接`。
6. GUI 展示识别到的任务列表。
7. 用户点击 `开始处理`。
8. 任务依次进入解析、下载、合并、提取音频、转写、写入文件。
9. 完成后用户点击 `打开校对稿` 或 `打开所在文件夹`。

### 7.2 抖音流程

1. 用户进入 `抖音` tab。
2. 粘贴 `v.douyin.com` 短链或分享文本。
3. 点击 `解析链接`。
4. GUI 展示任务列表。
5. 用户点击 `开始处理`。
6. 完成后在结果区打开或复制文案。

### 7.3 失败恢复流程

当任务失败时：

1. 任务卡显示失败原因的人类可读摘要。
2. 用户可展开查看原始日志。
3. 常见错误提供直接操作：
   - B 站 412：使用 Chrome/Edge cookies 重试。
   - ffmpeg 缺失：选择 ffmpeg.exe。
   - 输出目录不可写：重新选择目录。
   - Whisper 模型加载失败：切换模型或打开缓存目录。

## 8. 功能需求

### FR-1 平台 Tab

系统必须提供两个主 tab：

- `抖音`
- `Bilibili`

每个 tab 有独立输入框、解析按钮和平台专属配置。

验收标准：

- 用户可以在两个 tab 之间切换。
- Bilibili tab 显示 cookies 选项。
- 抖音 tab 不显示 B 站 cookies 配置。

### FR-2 多链接输入

系统必须支持粘贴一条或多条链接/分享文本，一行一个或混合在分享文本中。

验收标准：

- 能识别 `b23.tv`、`bilibili.com`、`bili2233.cn`。
- 能识别 `v.douyin.com`、`douyin.com`、`iesdouyin.com`、`amemv.com`。
- 重复链接只创建一个任务。
- 无效链接在解析结果中列出，不进入处理队列。

### FR-3 解析阶段

系统必须先解析链接，再进入处理。

验收标准：

- 点击 `解析链接` 后显示识别到的任务数量。
- 用户能在开始处理前清空或重新解析。
- 解析失败不触发下载。

### FR-4 任务队列

系统必须用任务列表展示每个视频的处理状态。

状态至少包括：

- 等待中
- 解析中
- 下载中
- 合并音视频
- 提取音频
- 转写中
- 写入文件
- 完成
- 失败
- 已取消

验收标准：

- 每个链接对应一个任务行或任务卡。
- 当前执行任务有高亮。
- 失败任务不会阻塞后续任务。

### FR-5 进度反馈

系统必须展示阶段级进度。

验收标准：

- 下载阶段显示百分比。
- 提取音频阶段显示当前阶段。
- 转写阶段显示 `转写中`、模型名和耗时。
- 完成时显示总耗时。

### FR-6 输出目录管理

系统必须允许用户选择输出目录，并记住上次选择。

验收标准：

- 默认输出目录为程序目录下 `outputs`。
- 用户可通过按钮选择目录。
- 用户可一键打开输出目录。
- 输出目录不可写时给出明确错误。

### FR-7 结果文件操作

系统必须展示每个完成任务的输出文件。

文件类型：

- Markdown 校对稿
- 原始 ASR 文本
- 视频文件
- 音频文件，如果保留
- info.json

验收标准：

- 每个文件提供 `打开`。
- 每个文件提供 `定位`。
- 每个文件提供 `复制路径`。
- 文件不存在时显示灰色不可用状态。

### FR-8 B 站 Cookies

系统必须允许用户为 B 站任务启用浏览器 cookies。

验收标准：

- 支持 `不使用`、`Chrome`、`Edge`。
- 默认不读取 cookies。
- 只有用户显式选择时读取 cookies。
- 读取 cookies 失败时显示可操作错误。

### FR-9 设置持久化

系统必须保存常用配置。

配置包括：

- 输出目录
- ffmpeg 路径
- Whisper 模型
- 是否保留 wav
- B 站 cookies 默认选择
- 窗口大小

验收标准：

- 关闭并重开应用后配置仍生效。
- 配置文件损坏时恢复默认值并提示。

### FR-10 日志详情

系统必须提供任务日志查看能力。

验收标准：

- 默认展示简短状态。
- 用户可展开查看详细日志。
- 可复制错误详情。
- 日志不泄露 cookies 内容。

## 9. 非功能需求

### NFR-1 响应性

GUI 主线程不得执行下载、ffmpeg、Whisper 转写等长任务。

验收标准：任务运行期间窗口可移动、可切换 tab、可打开设置。

### NFR-2 本地优先

V1 所有处理在本地完成，不上传视频、音频、cookies 或转写内容到远程服务。

### NFR-3 可维护性

CLI 与 GUI 必须共用核心服务层，避免复制处理逻辑。

### NFR-4 可打包性

项目必须能通过 PyInstaller 打包 Windows 桌面应用。

### NFR-5 安全性

读取浏览器 cookies 必须由用户显式开启。日志和 UI 不得输出 cookie 值。

## 10. 成功指标

- 用户无需命令行即可处理 B 站和抖音视频。
- B 站 412 场景能通过 Chrome cookies 重试成功。
- 用户可以在 GUI 中直接打开校对稿和输出目录。
- 批量处理时单个失败不影响其他任务。
- V1 完成后 CLI 仍保持可用。
